# Literature Tuna — 金融文档 RAG 系统

## 项目概述

从零构建的检索增强生成（RAG）系统，专为中文金融文档设计。用户可以用自然语言查询年报、研报、学术论文，系统检索相关内容后由 LLM 生成带来源引用的回答。无 LangChain/LlamaIndex 等黑盒框架，每个组件手写实现。

## 技术架构

### 数据流

```
输入文档（docx/pdf/txt）
        ↓
   文本提取（python-docx / pypdf）
        ↓
   中文金融分块器（章节感知）
        ↓
   向量嵌入（SiliconFlow BGE-large-zh）
        ↓
   numpy 矩阵存储（.npy + .json）
        ↓
   余弦相似度检索 → 上下文构建 → LLM 生成（DeepSeek-V3）
        ↓
   带来源引用的回答
```

### 核心设计决策

**1. 无框架路线**

不依赖 LangChain 或 LlamaIndex，手写全链路。设计意图有二：一是透彻理解 RAG 每一环节的原理，二是面试时可以清晰讲解每一行代码的用途。检索核心实质是一行 `np.dot` 余弦相似度，不存在抽象泄漏。

**2. 中文金融文档分块策略**

中文金融文档有独特的结构——使用「一、二、三、」作为章节标记。通用分块器（如 LangChain 的 RecursiveCharacterTextSplitter）无法识别这种结构。自研分块器先按章节标题分割，保留文档的层级关系；长章节在句子边界二次切分，加 50 字 overlap。分块质量直接影响召回率。

**3. 零向量数据库**

初期语料规模小（146 个块），用 ChromaDB 或 FAISS 属于过度工程。直接用 numpy 存储嵌入矩阵，查询时计算余弦相似度。系统简单、可控、无额外依赖，且策略可以随数据规模增长平滑升级。

**4. 嵌入与生成合一**

嵌入和生成共用 SiliconFlow API，统一鉴权。嵌入模型选 BGE-large-zh（中文检索专用，1024 维），生成模型选 DeepSeek-V3。无需维护多套 API Key。

## 技术栈

| 层级 | 选型 | 取舍说明 |
|------|------|---------|
| 文档解析 | python-docx + pypdf | 现有依赖，支持 docx/pdf/txt |
| 文本分块 | 自研章节感知分块 | 比通用分块器更适合中文金融文档 |
| 向量化 | BGE-large-zh（via SiliconFlow） | 中文检索优于 OpenAI ada-002 |
| 向量存储 | numpy .npy + JSON | 小规模够用，无须数据库 |
| 检索 | 余弦相似度（numpy） | 一行代码，面试可讲 |
| 生成 | DeepSeek-V3（via SiliconFlow） | 中文能力强，性价比高 |
| 接口 | 交互式 CLI + 单查询模式 | 匹配用户使用习惯 |

## 项目结构

```
├── rag_config.py    配置层：API 鉴权、路径、模型选择、分块参数
├── rag_utils.py     工具层：三种格式解析 + 中文金融分块器
├── rag_ingest.py    摄入管道：文档扫描 → 文本提取 → 分块 → 嵌入 → 持久化
├── rag_query.py     查询引擎：交互式 CLI / 单查询，检索 + 生成
├── rag_index.json   元数据：146 个文本块的来源、章节、内容
├── docs/            语料库：19 篇恒大案例金融新闻（公开来源）
├── .env             API Key 配置（不入库）
└── .gitignore       排除敏感文件
```

四条规则驱动文件拆分：配置集中（改参数不翻代码）、工具纯净（纯函数无状态）、摄入离线（一次性管道）、查询轻量（启动快秒级响应）。

## 设计取舍

### 不做的事

- 不用 LangChain/LlamaIndex：RAG 的核心逻辑并不复杂，框架带来的抽象成本超过收益
- 不用向量数据库：146 个块用 ChromaDB 是杀鸡用牛刀。numpy 在万块级别依然胜任，届时再迁移不迟
- 不用 Web UI：CLI 开发成本最低，演示效果足够。Streamlit 可在产品化阶段加入
- 不多轮对话：每问独立检索，避免历史污染。需要时在 prompt 层维护上下文即可

### 已知局限

- 分块器对 PDF 提取质量依赖 pypdf 自身能力，表格和复杂排版可能丢失信息
- 单轮检索不做重排序（rerank），top-k 中可能混入低相关段落
- 语料目前仅覆盖房地产/金融监管领域，扩展行业需补充文档

## 为什么这算 AI + 金融能力

AI 部分：LLM 应用落地、向量检索、Embedding 选型、Prompt Engineering、API 集成、Python 工程。

金融部分：理解金融文档结构（章节层级、披露格式）、熟悉中文金融术语、能用金融场景验证系统效果。

两者的结合才是这个项目的价值——不是做一个通用的 RAG demo，而是解决"中文金融文档怎么切、怎么搜、怎么答"这个特定问题。
